# Atelier — Package C (revised): AI-First Exercise Generation & Correction

**Supersedes `ATELIER_CONTENT_BANK_PLAN.md`.** We are NOT building a static bank or a
deterministic grammar engine as a gatekeeper. The product is AI-first: exercises are
generated fresh, per user, per session, from that learner's due concepts, recent
errors and due vocabulary — and corrections are written by an AI tutor, not templates.

## Philosophy

The LLM is the engine, not a suspect. We make it succeed by giving it enough context
and by having **AI check AI**, not by hardcoding French grammar in Python. Quality is
assured through:
- rich context + a coherent generation contract (most failures are context/contract
  bugs, not model stupidity), and
- an AI self-critique pass (LLM-as-judge) that catches gross errors and triggers
  regeneration.

The only non-AI guard is a thin **renderability/solvability** check (is the JSON
well-formed; can the answer actually be built from the offered chips) so the UI never
shows an unplayable item — and even that just triggers regeneration, it never rewrites
content. We do not deterministically "confirm grammar"; the AI judges grammar.

> Why "de bus de" happened: a contradictory contract — the schema forced
> `answer_tokens` to `minItems: 3` while the real answer was one token ("de"), so the
> model padded — compounded by generating the whole payload in one diluted call. That
> is a context/contract failure, fixable by A1 + A2, not by a code gatekeeper.

## Task A1 — Generation: full context + a coherent contract

Files: `app/services/exercise_generation.py` (system prompt ~389, schema),
`app/services/atelier.py` (`get_or_create` / `_generate_with_llm` → per-user generate).

- Feed the generator everything it needs to be *specific*, not generic:
  the concept rule/pattern/anchor examples; the learner's CEFR level; their recent
  error sub-patterns; their due vocabulary (so sentences use those words); and the
  **role of each mode** (word_bank = build ONE full sentence from scrambled chips,
  no blank; fill = single blank; classify = contrastive grammatical forms).
- Remove the contradictory schema constraint that caused padding (the blanket
  `minItems: 3` on `answer_tokens` for single-token answers) — make constraints
  mode-appropriate. Add one concrete worked example per mode in the prompt.
- Generate **per session, per user** (drop the shared-cache semantics of
  `get_or_create`). Consider focused per-mode calls so the model isn't juggling the
  whole payload at once; a well-structured single call is acceptable if quality holds.

Acceptance: regenerating the negation concept for a learner yields a coherent
full-sentence word-bank that uses their due vocab where natural, with a correct answer
key — no `___` chips, no padded tokens.

## Task A2 — AI self-critique (LLM checks LLM)

New `critique_exercise_payload(concept, payload, user) -> list[ItemVerdict]`, reusing
the correction infra (`_llm_correction`, `_correction_system_prompt`,
`ATELIER_CORRECTION_RESPONSE_FORMAT`).

- A fast/cheap model grades each item: is it solvable from the given chips/choices?
  does the answer key match? does it actually test the concept? is the French
  natural and correct? Returns per-item pass/fail + a one-line reason.
- Failed items are **regenerated with the critique fed back** (max ~2 passes). This
  replaces deterministic grammar gatekeeping entirely.
- Keep a thin structural/solvability guard (valid shape so it renders; answer buildable
  from chips so it's playable). On failure → regenerate, never hardcode-fix.

Acceptance: an injected `de bus de`-style payload is caught and regenerated by the
critic; clean payloads pass untouched; the guard never edits content, only re-requests.

## Task A3 — Latency: fast models + background pre-generation

Mini models are now strong enough — your default `OPENAI_MODEL` is already
`gpt-5-mini`. Use that tier; latency is a scheduling problem, not a reason to pre-bake.

- Config: keep `ATELIER_EXERCISE_LLM_MODEL` (a fast capable model for generation) and
  add `ATELIER_CRITIQUE_LLM_MODEL` (an even cheaper model for A2). Use
  `reasoning_effort` where supported.
- **Background pre-generation:** generate the learner's *next* session while they work
  the current one; cache the generated+critiqued session per user so session start is
  instant. Optionally stream item 1 immediately and fill the rest in the background.

Acceptance: first item visible in <2s on session start; the rest is ready before the
learner finishes item 1. No blocking spinner.

## Task A4 — AI-first error correction (make the LLM primary)

Files: `app/services/atelier.py` `correct()` (~2150), `_correct_recognize`,
`_correct_*_with_llm` (~2816–2922), config `ATELIER_CORRECTION_LLM_ENABLED`.

- The LLM correction paths already exist with rule-based templates as fallback. Make
  the **LLM the primary** corrector for every mode: given the learner's exact answer +
  the target + the concept, write a contextual explanation of the mistake and a
  targeted micro-correction. Demote the templates (`_word_bank_erratum` etc.) to the
  offline/unavailable fallback only.
- **Close the loop:** persist the learner's error sub-patterns and feed them into A1
  generation, so the next session's exercises target exactly what they got wrong. This
  is the AI-first personalization promise, end to end.

Acceptance: a wrong answer returns contextual, personalized feedback (never "Rebuild
the sentence as: {target}"); the missed sub-pattern measurably reappears in the next
generated session.

## Task A5 — Quality feedback loop (AI + light human)

- Log every generation + critique outcome (model, pass/fail, reasons) and any user
  "report this exercise" signal. Sample periodically to refine the prompts.
- No static bank, no manual catalog review — a continuous signal that improves the
  prompts and surfaces systematic failures per concept.

Acceptance: a queryable log/metric of generation and critique outcomes per concept;
a path to flag + regenerate a reported item.

## Dropped from the old plan (Package C bank version)
- `atelier_item_bank` table, the offline bank builder, and the deterministic grammar
  engine as a correctness gatekeeper. (A conjugation library may be used as *one
  optional signal inside the AI critique*, never as the blocking arbiter.)

## Kept (as resilience, not philosophy)
- The structural validator from the Fix Plan — reframed as the thin
  renderability/solvability guard that triggers regeneration.
- `_fallback_payload` as a last resort **only** if generation + critique fail
  repeatedly, so the app degrades gracefully instead of erroring.

## Definition of done
Every session is generated fresh for the individual learner from their
concepts/errors/vocab, judged by an AI critic, corrected by an AI tutor, served with
no perceptible latency — and there is no hardcoded grammar template acting as the
source of truth.
